# 在线部署与本地抓取方案教程

适用目标：
- 邮件滚动、点击、正文提取继续放在用户本地机器运行；
- 服务器只负责账号体系、验证码、模型调用、摘要历史、额度控制等；
- 普通用户不需要自己配置 OpenAI / DeepSeek API Key；
- 产品形态从“纯本地工具”升级为“本地助手 + 在线服务”。

==================================================
一、先说结论：你最终需要什么
==================================================

最小可上线版本通常需要：

1. 一台服务器
   用来跑后端 API、账号系统、验证码逻辑、模型调用。

2. 一个域名
   例如 `yourapp.com`，给网站和 API 用。

3. HTTPS 证书
   没有 HTTPS，登录、验证码、客户端请求都会不安全；浏览器和桌面端也更容易报不可信。

4. 发邮件能力
   用于注册验证码、重置密码、通知邮件。可以接第三方事务邮件服务。

5. 数据库
   保存用户、验证码、摘要历史、额度、日志等。前期可以 SQLite，正式上线更建议 PostgreSQL。

6. 模型 API Key
   放在服务器环境变量里，由服务器统一调用模型。

7. 一个本地客户端
   负责 Playwright 登录邮箱、滚动列表、提取正文，然后把待总结内容发给你的服务器。


==================================================
二、推荐的最终架构
==================================================

建议采用下面这条链路：

[用户本地客户端]
    |
    | 1. 用户登录学校邮箱
    | 2. 本地 Playwright 滚动、提取邮件正文
    | 3. 本地只把“需要总结的邮件文本 + 标题 + 日期 + 发件人”发给服务器
    v
[你的在线后端 API]
    |
    | 1. 校验用户登录态 / JWT / Session
    | 2. 扣减额度（如有）
    | 3. 调用 LLM
    | 4. 保存摘要历史
    v
[数据库 + 邮件服务 + 模型服务]

这套方案的好处：

- Playwright 不占服务器资源，避免多人在线时浏览器并发过差；
- 学校邮箱 Cookie 不需要上传到你的服务器，风险更低；
- 用户不用自己填 API Key，产品门槛低；
- 账号系统、验证码、计费/额度、历史记录都可以统一做。


==================================================
三、为什么这种方案比“全部放服务器”更合理
==================================================

你的当前项目里，`src/app.py` 维护了全局 `state`、全局浏览器对象和 `execute_lock`。
这说明当前后端天然更像“本地单人桌面服务”，而不是“高并发云端多租户服务”。

如果直接把现有 Playwright 部分搬到公网服务器，会遇到：

- 浏览器实例很吃内存和 CPU；
- OWA 页面滚动、等待、点击本身就慢；
- 多用户同时请求时容易排队；
- 还要在服务器上处理用户 Cookie、邮箱会话，风险更高。

所以更稳的路线是：

- 抓取继续留在本地；
- 云端只做轻量且可扩展的事。


==================================================
四、你需要准备的账号与资源
==================================================

建议先一次性准备这些：

1. 云服务器
   推荐先用 Linux 服务器，例如 Ubuntu 22.04。

2. 域名
   可以在阿里云、腾讯云、Cloudflare Registrar、Namecheap 等买。

3. DNS 托管
   推荐 Cloudflare 或域名商自带 DNS。

4. 事务邮件服务
   用于发验证码。常见方案：
   - Resend
   - SendGrid
   - 阿里云邮件推送
   - 腾讯云邮件推送

5. 模型供应商账号
   例如 OpenAI 兼容服务、DeepSeek、火山、硅基流动等。

6. 数据库
   前期：
   - 单机 VPS：SQLite 也能跑；
   中后期：
   - PostgreSQL 更适合正式上线。

7. 代码仓库
   最好把线上后端和本地客户端拆成两个目录或两个入口。


==================================================
五、你应该如何拆分当前项目
==================================================

建议把现在的能力拆成“两部分”：

A. 本地客户端
- 保留 Playwright、Cookie、OWA 选择器、正文提取；
- 主要复用或改造 `src/main.py` 里的提取能力；
- 抓完后不直接本地调 LLM，而是调用在线 API。

B. 云端服务
- 保留或改造 `src/app.py` 中的 Web 服务部分；
- 去掉“服务器直接登录用户邮箱并滚动抓取”的依赖；
- 增强成正式账号系统、验证码系统、历史记录系统、模型调用层。

可以这样理解：

- `main.py` 更像“本地邮件采集器”
- `app.py` 更像“云端摘要平台 API”


==================================================
六、推荐的实施顺序
==================================================

不要一口气做完所有功能，建议分四阶段。

--------------------------------------------------
阶段 1：做出最小可用在线版
--------------------------------------------------

目标：
- 用户可注册、登录；
- 本地客户端可把邮件文本发到服务器；
- 服务器能返回摘要；
- 用户不用配置 API Key。

这个阶段先不追求特别专业的功能，只要能闭环：

1. 用户在网站注册账号；
2. 本地客户端登录你的账号体系；
3. 本地抓取 3 到 10 封邮件；
4. 把邮件正文 POST 到服务器；
5. 服务器调用模型返回摘要；
6. 服务器保存一份摘要历史。


--------------------------------------------------
阶段 2：加入邮箱验证码与找回密码
--------------------------------------------------

目标：
- 注册时发邮箱验证码；
- 支持忘记密码；
- 提升“专业感”。

这一阶段要加入：

- 邮箱验证码发送接口；
- 验证码过期时间；
- 同一邮箱发送频率限制；
- 忘记密码邮件；
- 重置密码页面；
- 图形验证码或行为验证（防刷）。


--------------------------------------------------
阶段 3：加入额度控制与风控
--------------------------------------------------

目标：
- 控制模型成本；
- 防止恶意注册和滥用。

建议加入：

- 每日摘要次数上限；
- 每次上传的最大邮件数 / 最大字符数；
- 用户级别额度；
- IP 限流；
- 发验证码限流；
- 登录失败次数限制。


--------------------------------------------------
阶段 4：客户端打包与体验优化
--------------------------------------------------

目标：
- 让普通用户也会安装；
- 不要求用户会 Python。

可选路线：

1. Python 本地小助手
   优点：复用现有代码最多。
   缺点：分发体验一般。

2. Electron 桌面端
   优点：产品感强。
   缺点：体积大，重写 UI 成本更高。

3. 浏览器扩展
   优点：天然贴近邮箱网页。
   缺点：对 OWA DOM 变化更敏感，权限与上架更麻烦。

对你当前项目来说，最省力的通常是：

- 先做 Python 本地客户端；
- 后面如果产品化成功，再考虑 Electron。


==================================================
七、服务器购买与基础配置教程
==================================================

--------------------------------------------------
第 1 步：购买服务器
--------------------------------------------------

建议配置（早期个人项目）：

- 2 核 CPU
- 2GB 或 4GB 内存
- 40GB SSD
- Ubuntu 22.04

如果服务器只做：
- 账号系统
- 发验证码
- 调模型
- 存历史

那这个配置通常够起步。

注意：
- 如果以后你还想在服务器上跑很多后台任务、向量检索、图像处理，就再升配；
- 但如果 Playwright 留在本地，服务器压力会小很多。


--------------------------------------------------
第 2 步：购买域名
--------------------------------------------------

买域名后建议规划两个地址：

- 主站：`app.yourdomain.com`
- API：`api.yourdomain.com`

也可以先偷懒只用一个：

- `yourdomain.com`

但后期分前后端时，分子域名会更清晰。


--------------------------------------------------
第 3 步：解析 DNS
--------------------------------------------------

在 DNS 里加：

- `A` 记录：`app.yourdomain.com` -> 服务器公网 IP
- `A` 记录：`api.yourdomain.com` -> 服务器公网 IP

如果使用 Cloudflare：
- 前期可以开启代理；
- 但某些 API 调试阶段建议先关闭代理，等服务稳定再打开。


--------------------------------------------------
第 4 步：登录服务器并安装基础环境
--------------------------------------------------

通常要安装：

- git
- python3
- python3-venv
- pip
- nginx 或 caddy

如果后面要 PostgreSQL，再装：

- postgresql
- postgresql-contrib


==================================================
八、后端上线教程（FastAPI）
==================================================

--------------------------------------------------
第 1 步：拉代码
--------------------------------------------------

在服务器上创建部署目录，例如：

- `/srv/xjtlu-mail-app`

然后拉取你的仓库代码。


--------------------------------------------------
第 2 步：建立虚拟环境并安装依赖
--------------------------------------------------

建议在服务器上创建独立虚拟环境：

- `python -m venv .venv`
- 激活虚拟环境
- `pip install -r src/requirements.txt`

如果后端不再直接跑 Playwright 抓邮箱，可以逐步把服务器端对 Playwright 的依赖变成“可选”，减少部署复杂度。


--------------------------------------------------
第 3 步：准备环境变量
--------------------------------------------------

线上至少要有这些变量：

- `SESSION_SECRET`
  用于登录会话签名，必须随机、够长。

- `OPENAI_API_KEY` 或你的模型服务 Key
  放服务器，不给用户。

- `OPENAI_BASE_URL`
  如使用兼容接口时需要。

- `OPENAI_MODEL`
  默认模型名。

- `MAIL_API_KEY` / `SMTP_USER` / `SMTP_PASS`
  验证码邮件所需。

- `APP_ENV=prod`
  可选，用于区分环境。


--------------------------------------------------
第 4 步：跑通 Uvicorn
--------------------------------------------------

本地测试通了之后，线上一般不会直接裸跑开发命令。

推荐至少用：

- `uvicorn app:app --host 127.0.0.1 --port 8001`

如果 `app.py` 在 `src/` 下，就在正确工作目录启动。

等反向代理配置好后，再由 Nginx/Caddy 对外暴露 80/443。


--------------------------------------------------
第 5 步：用 systemd 托管进程
--------------------------------------------------

目标：
- 开机自启；
- 崩溃自动拉起；
- 可统一看日志。

systemd 服务文件通常要指定：

- WorkingDirectory
- ExecStart
- User
- Environment
- Restart=always

上线后不要靠“手动开一个终端运行 Python”。


==================================================
九、域名与 HTTPS 教程
==================================================

--------------------------------------------------
第 1 步：配置反向代理
--------------------------------------------------

推荐：

- Nginx：资料多、最常见；
- Caddy：自动 HTTPS 很省心。

基本思路：

- 用户访问 `https://app.yourdomain.com`
- 反向代理转发到 `127.0.0.1:8001`


--------------------------------------------------
第 2 步：申请 HTTPS 证书
--------------------------------------------------

最常见方式：

- Let’s Encrypt

如果用 Caddy，很多时候会自动签发和续期。
如果用 Nginx，常见搭配是 certbot。

上线后一定要确认：

- 浏览器访问站点不报不安全；
- API 请求走 `https://`；
- 验证码和登录只走 HTTPS。


==================================================
十、验证码邮件教程
==================================================

--------------------------------------------------
第 1 步：选择发信方式
--------------------------------------------------

推荐直接用事务邮件服务，不建议一开始自建邮局。

常见可选：

- Resend
- SendGrid
- 阿里云邮件推送
- 腾讯云邮件推送


--------------------------------------------------
第 2 步：配置域名邮件认证
--------------------------------------------------

若你用自定义域名发验证码，例如：

- `no-reply@yourdomain.com`

通常要在 DNS 添加：

- SPF
- DKIM
- DMARC（建议）

否则验证码邮件很容易进垃圾箱。


--------------------------------------------------
第 3 步：设计验证码逻辑
--------------------------------------------------

建议规则：

- 验证码 6 位数字；
- 10 分钟过期；
- 同一邮箱 60 秒内不能重复发送；
- 每日发送上限；
- 注册、重置密码分别记录用途；
- 验证成功后立刻失效。

数据库里至少要记录：

- email
- code_hash 或 code
- purpose（register / reset_password）
- expire_at
- created_at
- consumed_at

更安全的做法是只存 hash，不存明文验证码。


==================================================
十一、数据库怎么选
==================================================

--------------------------------------------------
方案 A：前期继续 SQLite
--------------------------------------------------

适合：
- 用户少；
- 单机部署；
- 先快速验证产品。

优点：
- 上手快；
- 迁移成本低；
- 你现在项目已经在用。

缺点：
- 多实例扩展差；
- 并发写入能力一般；
- 做线上正式产品时心里不踏实。


--------------------------------------------------
方案 B：迁移 PostgreSQL
--------------------------------------------------

适合：
- 真的要长期上线；
- 可能以后分服务、分实例；
- 想做更标准的线上架构。

建议：
- 如果只是毕业设计 / MVP，可先 SQLite；
- 如果准备对外运营，尽早迁 PostgreSQL。


==================================================
十二、本地客户端应该怎么做
==================================================

本地客户端的职责尽量收敛，不要又抓邮箱又当一套完整 Web 服务。

建议它只做四件事：

1. 登录你的产品账号
   获取你的服务端 token / session。

2. 连接学校邮箱
   通过 Cookie 或交互式登录。

3. 抓取并整理邮件
   输出结构化数据，例如：
   - subject
   - sender
   - date
   - body

4. 调用云端总结接口
   把结构化邮件列表发给服务器，拿回摘要。

客户端不要做的事：

- 不要直接保存你的模型 Key；
- 不要做太多复杂业务规则；
- 不要让前后端业务逻辑在客户端和云端重复维护两套。


==================================================
十三、在线 API 应该怎么设计
==================================================

建议最少有这些接口：

1. 账号相关
- `POST /api/auth/register`
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

2. 验证码相关
- `POST /api/auth/send-code`
- `POST /api/auth/verify-code`
- `POST /api/auth/forgot-password`
- `POST /api/auth/reset-password`

3. 总结相关
- `POST /api/summaries`
  输入：邮件数组、用户指令
  输出：摘要文本、tokens、耗时、摘要记录 ID

- `GET /api/summaries/history`
  查询历史记录

4. 额度相关（可选）
- `GET /api/quota`

如果以后要支持“本地客户端登录”，可以增加：

- `POST /api/client/login`
- `POST /api/client/refresh-token`


==================================================
十四、上线前的安全清单
==================================================

至少要做到：

1. 全站 HTTPS
2. 密码使用 bcrypt 或 argon2 哈希
3. 会话密钥不要写进仓库
4. 注册 / 登录 / 发验证码接口限流
5. API Key 只放服务器
6. 不在服务器保存学校邮箱 Cookie（或尽量不保存）
7. 数据库定期备份
8. 记录错误日志，但日志里不要打印敏感内容
9. 说明隐私政策：邮件正文什么时候上传、保存多久、是否用于训练


==================================================
十五、成本与现实预估
==================================================

一个最小可用版本的大头成本通常是：

1. 服务器
2. 域名
3. 事务邮件服务
4. 模型调用费用

如果用户真的开始使用，模型费通常会比服务器费更快上涨。

所以一定要尽早加入：

- 单次最大邮件数限制；
- 单次最大正文长度限制；
- 每日免费额度；
- 用户级限流。


==================================================
十六、和你当前项目最贴近的迁移路线
==================================================

建议按下面做，不容易把现有项目搞崩：

第 1 步：
- 保留现有 `src/main.py` 的邮件提取能力；
- 先在本地写一个“调用云端总结”的新函数；
- 证明“本地抓取 -> 云端总结”链路能跑通。

第 2 步：
- 保留现有 `src/app.py` 的账号、页面框架；
- 新增真正的线上环境变量、域名、HTTPS。

第 3 步：
- 增加验证码表、邮箱发送接口、重置密码流程；
- 把账号体系从“本地工具登录”升级为“线上产品登录”。

第 4 步：
- 把本地抓取部分从当前 Web 服务里拆出来，变成独立本地客户端；
- 线上只保留纯后端摘要平台。


==================================================
十七、你现在最应该先做什么
==================================================

如果你不想一下子做太大，建议立刻只做下面 3 件：

1. 先确认产品形态
   明确采用：
   “本地抓取 + 云端总结 + 服务端统一 API Key”

2. 先买最小资源
   - 1 台 Ubuntu 云服务器
   - 1 个域名
   - 1 个事务邮件服务账号

3. 先跑最小闭环
   - 用户注册 / 登录
   - 本地发送 1 封测试邮件文本给服务器
   - 服务器返回摘要

只要这条链路通了，后面的验证码、历史、额度、客户端打包都只是持续迭代。


==================================================
十八、最后的建议
==================================================

你现在最合理的定位不是：

- “把当前项目原样扔上云”

而是：

- “把当前项目拆成：本地采集器 + 在线摘要平台”

这是从“课程项目 / 本地工具”走向“可上线产品”的关键一步。

如果你后续要我继续帮你推进，下一步最合适的是二选一：

1. 我帮你写一份“接口设计文档”
   重点写：本地客户端怎么把邮件发给云端、云端怎么回摘要。

2. 我帮你写一份“部署清单 + 上线顺序”
   重点写：先改哪些文件、数据库怎么加验证码表、域名和邮件服务怎么接。
