# 从个人工具到校友产品的功能优先级计划

## 当前判断

现有版本已经完成了“个人可用”的核心闭环：搜索邮件、提取正文、调用 LLM 总结，主流程集中在 [src/app.py](src/app.py)、[src/main.py](src/main.py) 和 [src/templates/index.html](src/templates/index.html)。

但如果目标变成“在线产品，校友普遍愿意使用”，当前最大的短板不是总结能力，而是：

- 上手门槛高：用户仍需理解 `Cookie`、`API Key`、浏览器自动化。
- 规模化与隔离仍不完整：账号与配置已按用户落库（SQLite），但浏览器资源池、任务队列与多实例部署仍偏“单机思维”，高并发与严格多租户隔离仍需演进。
- 信任感不足：账号数据、邮箱内容、AI 调用成本与隐私边界还需要更明确的产品化表达（文案与流程）。
- 留存价值不足：当前主价值是“单次总结”，还没有形成持续回访理由。

## 优先级建议

### P1 先做“能放心开始用”

这是最优先级，决定有没有人愿意第一次尝试。

目标：把产品从“技术同学才能配置”改成“普通校友也能完成首次使用”。

建议优先做：

- 账号体系与个人工作区：至少区分不同用户的任务、配置和结果。（已完成）
- 安全的连接邮箱流程：把现在直接暴露在界面的 `Cookie` 粘贴流程，改成更可理解的“连接邮箱”向导。
- 托管式 AI 配置：不要要求普通用户自己填 `API Key`，否则产品会卡死在极客用户层。
- 明确隐私说明：在页面中明确“读取什么、保存多久、是否可删除、是否会上传给模型”。
- 任务状态与失败提示产品化：当前已有 `/api/status` 基础能力，可升级为可见的任务中心与重试机制。

这是第一阶段，因为当前 [src/templates/index.html](src/templates/index.html) 仍把配置、Cookie、执行都放在同一操作台，适合自己用，不适合大范围传播。

### P2 再做“为什么我要持续用”

当使用门槛下降后，下一步不是堆更多按钮，而是增强持续价值。

建议优先做：

- 自动分类视图：把邮件分成 `活动`、`招聘/实习`、`课程/学习`、`校友通知`、`待办`。
- 每日/每周 Digest：不只按关键词单次运行，而是自动生成固定节奏摘要。
- 重要邮件提醒：识别 deadline、报名截止、付款、会议等高风险事项。
- 个性化偏好：让用户勾选“我更关心招聘”“我只看活动”“英文邮件自动翻中文”。
- 历史可回看：保存往期摘要和关键事项，形成个人信息中台。

原因：当前能力本质上是“按次查询”，而校友产品更需要“持续帮我过滤信息”。

### P3 最后做“校友之间会愿意推荐”

当产品已经易用且有留存理由，再补传播性功能。

建议优先做：

- 校友场景模板：例如 `求职季邮件模式`、`活动报名模式`、`毕业手续模式`。
- 群体化摘要：面向社群管理员生成“本周值得看的 5 封邮件”。
- 可分享的摘要卡片或链接：便于在微信群、朋友圈、校友群传播。
- 用户反馈闭环：让用户标记“这封有用/没用”，反向训练优先级和摘要风格。

这类功能不应放在第一阶段，因为如果前两阶段没打稳，传播会先放大产品摩擦。

## 不建议现在优先做的事

- 不建议先追求更复杂的 UI 重做。当前问题主要不是视觉，而是产品路径与信任模型。
- 不建议先加更多高级提示词玩法。普通用户不会因为提示词更自由就更愿意留下来。
- 不建议先支持太多邮箱来源。先把西浦邮箱场景做深，比过早泛化更重要。

## 结合当前代码的落地方向

- [src/app.py](src/app.py) + [src/auth_db.py](src/auth_db.py)：配置已按用户存储；后续重点是把任务执行、浏览器占用与失败重试做成可观测、可排队的服务化能力。
- [src/main.py](src/main.py)：可保留邮件抓取与总结能力，但要拆成更适合服务端调度的任务单元。
- [src/templates/index.html](src/templates/index.html) 与 [src/templates/login.html](src/templates/login.html) / [src/templates/register.html](src/templates/register.html)：下一阶段主界面应从“工程控制台”转成“连接邮箱 + 查看摘要 + 管理偏好”的清晰路径。
- [README.md](README.md)：后续也要同步改写定位，从“自动化工具说明”升级为“产品价值与使用路径说明”。

## 建议的执行顺序

1. 先补用户接入和信任基础：账号、隐私说明、连接邮箱、托管模型配置。
2. 再补留存能力：自动分类、Digest、提醒、历史记录。
3. 最后补传播能力：模板、分享、反馈闭环。

如果你接受这个方向，下一步最值得细化的是第一阶段，我会把它进一步拆成“具体要改哪些页面、接口、数据结构，以及最小可上线版本怎么做”。
